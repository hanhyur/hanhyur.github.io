# HTTP 메서드

HTTP 메서드에 대해서 알아봅시다. HTTP 메서드 종류는 기본적으로 GET, POST, PUT, PATCH, DELETE가 있습니다. 각 메서드의 세부 내용을 살펴보기 전에 간단한 HTTP API를 만들어봅시다.

## HTTP API를 만들어보자.

실무에 투입이 되었다고 가정해보겠습니다. '회원 정보 관리 API를 만들어라'는 요구사항이 주어졌을 때 다음과 같은 기능을 생각할 수 있을 것입니다.

- 회원 목록 조회
- 회원 조회
- 회원 등록
- 회원 수정
- 회원 삭제

자 그러면 여기서 URI를 만들려면 어떻게 만들어야 할까요? 개발을 접한지 얼마 되지 않았다면 다음과 같이 생각할 수 있습니다.

- 회원 목록 조회 /read-member-list
- 회원 조회 /read-member-by-id
- 회원 등록 /create-member
- 회원 수정 /update-member
- 회원 삭제 /delete-member

그렇다면 과연 이것은 좋은 URI 설계일까요? 한번쯤 고민해봐야 합니다.  
URI를 잘 설계하기 위해 가장 중요한 것은 <b>리소스의 식별</b>입니다. 리소스가 무엇일까요?

리소스는 회원을 등록하고 수정하고 조회하는 것이 아닙니다. 예를 들어 LoL에서 챔피언이 CS를 먹어라고 명령을 내린다고 생각해봅시다. 여기서 'CS를 먹어!'에서 리소스는 CS입니다. 즉, 요구사항에서의 리소스는 <b>회원</b>이라는 개념 자체입니다.

그렇다면 리소스를 어떻게 식별하는게 좋을까요? 회원으로 하는 행동, 즉 등록, 수정, 조회하는 것을 배제하는 것입니다. 회원이라는 리소스만 식별해서 URI에 매핑하면 됩니다. 실제로 적용하게 되면 다음과 같이 될 수 있습니다.

- 회원 목록 조회 /members
- 회원 조회 /members/{id}
- 회원 등록 /members/{id}
- 회원 수정 /members/{id}
- 회원 삭제 /members/{id}

리소스를 식별하고 URI의 계층 구조를 사용한 것은 좋은데 조회, 등록, 수정, 삭제가 동일한 URI를 가지고 있습니다. 어떻게 해야할까요? 여기서 메서드가 사용됩니다.

### 리소스와 행위를 분리

URI는 리소스만 식별해야 합니다. 그래서 리소스와 해당 리소스를 대상으로 하는 <b>행위</b>를 분리하면 다음과 같습니다.

- 리소스 : 회원
- 행위 : 조회, 등록, 삭제, 변경

보통 리소스는 명사가 되고, 행위는 동사가 됩니다. 그렇다면 이 행위(메서드)는 어떻게 구분할까요? 바로 HTTP 메서드가 그 역할을 해줍니다.
